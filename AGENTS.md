# AGENTS.md

このリポジトリは、寮の共同洗濯機の空き状況可視化・利用開始登録・通知導線を提供するフロントエンドです。
エージェントは以下の指示に従って作業してください。

---

## 目的

このプロダクトの目的は次の通りです。

- 共同洗濯機の状態を分かりやすく表示する
- 「行ってみたら埋まっていた」という無駄足を減らす
- 利用開始登録と回収完了導線をシンプルにする
- PWAとしてスマホで使いやすくする
- 個人情報をできるだけ持たないUIにする

このリポジトリはフロントエンド専用です。
バックエンドの業務ロジックをフロント側に埋め込まないでください。

---

## 技術スタック

このリポジトリでは以下を前提にしてください。

- Next.js
- TypeScript
- App Router
- Tailwind CSS
- PWA対応
- TanStack Query
- 必要に応じて Zod
- 必要に応じて React Hook Form

技術選定は勝手に変更しないでください。
別ライブラリの導入が必要な場合は、既存構成との整合性を最優先にしてください。

- コメント、説明、PR用要約、作業報告も日本語

## 実装方針

### 最優先
- モバイルで見やすい
- 状態が一目で分かる
- 誤操作しにくい
- 過剰に装飾しない
- 軽い
- 将来のAPI差し替えに強い

### UI方針
- UIはミニマルにする
- 派手なアニメーションは不要
- 色だけに依存して状態を伝えない
- 状態表示には文字ラベルを必ず併用する
- 1画面でやることを詰め込みすぎない

### 情報設計
利用者向け画面は次を優先してください。

1. 洗濯機一覧
2. 各洗濯機の状態
3. 利用開始
4. 自分の利用中セッション確認
5. 回収完了
6. 通知許可導線

管理機能は最小限にしてください。

---

## 状態定義

UIで扱う状態は原則として以下です。

- `available`
- `running`
- `awaiting_pickup`
- `offline`

表示文言は例えば以下のようにしてください。

- available: 空き
- running: 使用中
- awaiting_pickup: 回収待ち
- offline: 利用不可

状態名の追加や変更は、型定義と表示文言の両方を必ず揃えてください。

---

## 画面一覧

最低限、以下の画面を想定してください。

- `/`
  - 洗濯機一覧画面
- `/machines/[machineId]`
  - 洗濯機詳細画面
- `/sessions/current`
  - 自分の現在の利用状況
- `/settings/notifications`
  - 通知設定
- `/admin`
  - 管理用画面（必要最低限）

新しい画面を増やす前に、既存画面へ統合できないかを検討してください。

---

## コンポーネント設計

### 推奨
- 表示とロジックを分離する
- 再利用可能なUIを `components/` に置く
- 画面固有のものは route 配下に閉じる
- APIレスポンスをそのままUIで使わず、必要に応じて整形する

### 避けること
- 巨大な page.tsx を作る
- 1ファイルにロジックを詰め込む
- props drilling が深くなりすぎる構成
- any の多用
- サーバー状態とUI状態の混同

# API連携方針
APIクライアントは lib/api/ に集約する
fetch処理を各コンポーネントに直書きしない
エンドポイントは定数化する
モック可能な設計にする
API未完成時はモックデータで画面開発可能にする

例:

GET /machines
GET /machines/:id
POST /sessions
POST /sessions/:id/claim
POST /push-subscriptions

HTTPエラー時は必ずユーザーに意味のある文言を出してください。
技術的エラーメッセージをそのまま表示しないでください。

# PWA方針

このプロダクトはPWA前提です。

ホーム画面追加を意識したUIにする
モバイルでの表示を最優先する
オフライン完全対応は初期段階では不要
ただし最低限のシェル表示や再訪問体験は壊さない
通知許可導線を自然に設計する

PWA実装のために複雑すぎるキャッシュ戦略を持ち込まないでください。

# 通知導線

通知は重要機能です。
ただし、初回表示で強制的に許可を求めないでください。

推奨フロー:

利用開始時または利用中画面で通知の価値を説明
明示的な操作の後に許可を促す
拒否された場合もアプリが破綻しないようにする

通知未許可ユーザーにも使えるUIを維持してください。

# 個人情報・プライバシー

このプロダクトでは、個人情報を最小限に扱ってください。

禁止
氏名を前提にしたUI
部屋番号を一覧公開するUI
他人の利用履歴を見せるUI
不要な識別子の長期保存を前提にした実装
原則
公開するのは洗濯機の状態のみ
自分のセッションだけ参照できればよい
所有者情報を見せなくても成立するUIを優先する
デザイン原則
スマホ縦画面を基準にする
タップ領域は狭くしない
余白を十分に取る
1画面1目的を意識する
状態カード中心で組む
ボタン文言は動詞で明確にする
例: 「利用開始」「回収完了」「通知を有効にする」
アクセシビリティ
状態を色だけで表現しない
ボタンには明確なラベルを付ける
読み上げで意味が崩れないようにする
ローディング、成功、失敗をUIで分かるようにする
コントラストを下げすぎない
ローディング・エラー・空状態

すべての主要画面で以下を必ず考慮してください。

loading
error
empty

特に洗濯機一覧では、通信失敗時に再読み込み導線を置いてください。

# テスト方針

最低限、以下を優先してください。

主要コンポーネントの表示テスト
状態表示の分岐
フォーム送信の基本動作
APIエラー時のUI表示

スナップショットテストだけに依存しないでください。

# 禁止事項
技術スタックの無断変更
CSSフレームワークの追加導入
UIライブラリの大量導入
不要な状態管理ライブラリの追加
個人情報を前提にした仕様追加
複雑すぎる抽象化
使っていないコンポーネントや hooks の量産
作業時のルール

変更時は次を意識してください。

変更理由が説明できること
既存構成との整合性があること
画面の責務が明確であること
将来バックエンドAPIに接続しやすいこと
モバイルUIを壊さないこと

大きな変更をするときは、以下の順で検討してください。

既存コンポーネントの再利用
features 単位での責務分離
新規コンポーネント作成
まず実装すべきもの

優先順位は以下です。

洗濯機一覧画面
洗濯機状態カード
利用開始フロー
利用中セッション画面
回収完了フロー
通知許可導線
管理画面の最小実装
期待する完成像

完成像は「高機能なランドリーアプリ」ではなく、まずは次を満たすものです。

寮生がスマホで空き状況を確認できる
利用開始登録がすぐできる
自分の洗濯終了を把握しやすい
回収完了までの導線が分かりやすい
UIが軽く、迷わない

余計な機能より、上記の確実な達成を優先してください。
